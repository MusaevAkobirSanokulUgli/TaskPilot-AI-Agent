🎯 Problem Statement

Why reactive chatbots fail at complex project management

Traditional chatbots are stateless and reactive — they answer one question at a time without memory, planning, or the ability to take multi-step actions. Project management involves coordination across tasks, timelines, priorities, and team members that requires persistent state and autonomous decision-making.

  • Chatbots cannot plan: they respond to individual prompts but cannot decompose a high-level goal into a sequence of executable steps.
  • No persistent memory: each conversation starts fresh, requiring users to repeat context that an agent should remember from previous sessions.
  • No tool access: chatbots cannot actually create tasks, query databases, generate reports, or send notifications — they can only describe how to do these things.
  • No self-correction: if an action fails, a simple chatbot cannot observe the failure, reason about it, and try a different approach automatically.
  • Mobile and real-time gap: project management happens across devices and requires push notifications and live updates — features absent from API-only chatbots.

1. Reason

Analyse the goal using LLM (GPT-4o-mini) or keyword heuristics to understand intent and constraints.

2. Plan

Decompose the goal into an ordered list of tool calls with expected parameters and fallback steps.

3. Act

Execute the next planned tool call — TaskManagerTool, CodeAnalyzerTool, ReportGeneratorTool, etc.

4. Observe

Capture the tool's result or error. Stream the observation event via SignalR to connected clients.

5. Remember

Update short-term, working, and long-term memory tiers with new observations and intermediate results.

6. Check

Evaluate whether the goal is achieved. If not, loop back to Reason with updated context. If yes, return result.

🧠 3-Tier Memory System

Hierarchical memory architecture enabling contextual continuity across agent sessions

Short-term Memory

20 entries

In-memory sliding window of recent observations and actions within the current ReAct loop iteration. Oldest entries evicted when capacity is reached.

Working Memory

Unbounded

All intermediate results, tool outputs, and reasoning traces for the current task. Scoped to a single task execution — cleared when the task completes.

Long-term Memory

500 entries

Persistent storage of completed task outcomes, learned patterns, and embeddings. Enables the agent to retrieve relevant past experience for new tasks via semantic similarity search.

✨ Key Features

Autonomous task management with full-stack real-time capabilities

🤖
6-Phase ReAct Loop
Reason → Plan → Act → Observe → Remember → Check. The agent autonomously iterates until the goal is achieved or a configurable iteration limit is reached.
🧠
Dual-mode Planning
LLM-based planning (GPT-4o-mini) when an API key is provided, falling back to rule-based keyword heuristics for zero-cost local operation.
📡
Real-time Event Streaming
Every agent event (reasoning step, tool call, observation) is streamed via SignalR WebSocket to the web dashboard in real time.
📱
Mobile Companion App
React Native / Expo app with push notifications for task assignment, status changes, and agent completion events on iOS and Android.
📌
Kanban Board
Drag-and-drop Kanban board with columns for Backlog, In Progress, Review, and Done. 8 seeded demo tasks with varied statuses and priorities.
🌱
Demo Seed Data
8 pre-seeded tasks covering diverse priorities (Critical, High, Medium, Low) and statuses to demonstrate the full project lifecycle on first launch.

🚀 Quick Start

Run TaskPilot with demo data — LLM is optional, rule-based mode works out of the box

Step 1: Clone the repository

git clone https://github.com/your-org/ai-agent-taskpilot.git cd ai-agent-taskpilot

Step 2: Configure environment (OpenAI key is optional)

cp .env.example .env # Optional: add your OpenAI key for LLM-based planning OPENAI_API_KEY=sk-... # Leave blank to use rule-based fallback AGENT_MAX_ITERATIONS=10 MEMORY_LONG_TERM_CAPACITY=500

Step 3: Start all services with Docker Compose

docker compose up --build -d # Services starting: # ✓ web-dashboard → http://localhost:3002 # ✓ dotnet-api → http://localhost:5002 # ✓ agent-core → http://localhost:8001

Step 4: Run an agent task via API

curl -X POST http://localhost:5002/api/agent/run \ -H "Content-Type: application/json" \ -d '{"goal": "Create a task for implementing user authentication and assign it high priority"}' # Returns: {"sessionId": "sess_abc", "status": "running"} # Watch events stream at ws://localhost:5002/hubs/agent

Step 5: Open the Kanban board at http://localhost:3002

🔒 Security Considerations

Safe autonomous agent operation with configurable guardrails

🔑 API Key Security

OpenAI API keys stored only in environment variables. The Python agent core never logs keys. Requests to OpenAI are made server-side only — keys are never exposed to browser clients.

🔁 Iteration Limits

The ReAct loop has a configurable maximum iteration count (default 10). This prevents infinite loops and runaway token costs when using LLM-based planning.

🛡 Tool Access Control

Each agent tool validates its inputs before execution. TaskManagerTool enforces field-level validation. WebSearcherTool has domain allow/block lists.

📡 WebSocket Authentication

SignalR hub connections require a valid session token. Unauthenticated clients receive only public events; task-level events require authorization.

💾 Memory Privacy

Long-term memory entries are scoped per user session. Embeddings for sensitive tasks are stored locally without sending to external embedding services.

📱 Mobile Push Notifications

Expo push tokens are stored encrypted. Notification payloads contain only task IDs — full task data is fetched by the client after receiving the notification.
