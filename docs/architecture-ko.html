🎯 문제 정의

기존 챗봇이 복잡한 프로젝트 관리에서 실패하는 이유

기존 챗봇은 상태가 없고 반응적입니다 — 메모리나 계획 없이, 다단계 행동 능력 없이 한 번에 하나의 질문에만 답합니다. 프로젝트 관리는 작업, 일정, 우선순위, 팀원 간의 조율이 필요하며, 이는 지속적인 상태와 자율적인 의사결정을 요구합니다.

  • 챗봇은 계획을 세울 수 없습니다: 개별 프롬프트에 응답하지만 고수준 목표를 실행 가능한 단계의 연속으로 분해할 수 없습니다.
  • 영속적인 메모리 없음: 각 대화는 새로 시작되어, 에이전트가 이전 세션에서 기억해야 할 컨텍스트를 사용자가 반복해야 합니다.
  • 도구 접근 불가: 챗봇은 실제로 작업을 생성하거나, 데이터베이스에 쿼리하거나, 보고서를 생성하거나, 알림을 보낼 수 없습니다 — 이러한 것들을 설명만 할 수 있습니다.
  • 자기 수정 불가: 행동이 실패하면, 단순한 챗봇은 실패를 관찰하고, 이에 대해 추론하며, 자동으로 다른 접근 방식을 시도할 수 없습니다.
  • 모바일 및 실시간 격차: 프로젝트 관리는 여러 기기에서 발생하며 푸시 알림과 라이브 업데이트가 필요합니다 — API 전용 챗봇에는 없는 기능들입니다.

1. 추론 (Reason)

LLM(GPT-4o-mini) 또는 키워드 휴리스틱을 사용하여 목표를 분석하고 의도와 제약을 이해합니다.

2. 계획 (Plan)

목표를 예상 매개변수와 폴백 단계가 있는 도구 호출의 순서 목록으로 분해합니다.

3. 행동 (Act)

다음 계획된 도구 호출을 실행합니다 — TaskManagerTool, CodeAnalyzerTool, ReportGeneratorTool 등.

4. 관찰 (Observe)

도구의 결과나 오류를 캡처합니다. SignalR을 통해 연결된 클라이언트에 관찰 이벤트를 스트리밍합니다.

5. 기억 (Remember)

새로운 관찰과 중간 결과로 단기, 작업 및 장기 메모리 계층을 업데이트합니다.

6. 확인 (Check)

목표가 달성되었는지 평가합니다. 그렇지 않으면 업데이트된 컨텍스트로 추론으로 돌아갑니다. 달성되었으면 결과를 반환합니다.

🧠 3단계 메모리 시스템

에이전트 세션 전반에 걸쳐 문맥적 연속성을 가능하게 하는 계층적 메모리 아키텍처

단기 메모리

20개 항목

현재 ReAct 루프 반복 내의 최근 관찰과 행동의 인메모리 슬라이딩 윈도우. 용량 초과 시 가장 오래된 항목이 제거됩니다.

작업 메모리

무제한

현재 작업의 모든 중간 결과, 도구 출력 및 추론 트레이스. 단일 작업 실행으로 범위가 제한되며 작업 완료 시 초기화됩니다.

장기 메모리

500개 항목

완료된 작업 결과, 학습된 패턴 및 임베딩의 영구 저장. 에이전트가 의미론적 유사성 검색을 통해 새 작업에 관련 과거 경험을 검색할 수 있게 합니다.

🔧 에이전트 도구

작업 실행 중 ReAct 에이전트가 사용할 수 있는 5가지 도구

📋
TaskManagerTool
프로젝트 관리 데이터베이스에서 작업을 생성, 업데이트, 삭제 및 쿼리합니다. 상태 전환 및 우선순위 변경을 지원합니다.
🔍
CodeAnalyzerTool
코드 스니펫이나 저장소 구조를 분석합니다. 복잡성을 추정하고, 패턴을 식별하며, 구조화된 요약을 생성합니다.
📊
ReportGeneratorTool
작업 데이터에서 Markdown 또는 JSON 형식으로 진행 보고서, 스프린트 요약 및 상태 업데이트를 생성합니다.
🌐
WebSearcherTool
작업 계획 결정에 정보를 제공하기 위해 기술 문서, 모범 사례 또는 관련 리소스를 검색합니다.
🔔
NotificationTool
Expo를 통해 모바일 기기에 푸시 알림을 보내고, SignalR WebSocket 허브를 통해 웹 클라이언트에 실시간 이벤트를 브로드캐스트합니다.

API 키가 제공될 때 LLM 기반 계획(GPT-4o-mini), 비용 없는 로컬 운영을 위한 규칙 기반 키워드 휴리스틱으로 폴백.

칸반 보드
백로그, 진행 중, 검토, 완료 열이 있는 드래그 앤 드롭 칸반 보드. 다양한 상태와 우선순위를 가진 8개의 시드된 데모 작업.
🌱
데모 시드 데이터
첫 시작 시 전체 프로젝트 수명 주기를 보여주기 위한 다양한 우선순위(Critical, High, Medium, Low)와 상태를 가진 8개의 사전 시드된 작업.

🚀 빠른 시작

데모 데이터로 TaskPilot 실행 — LLM은 선택 사항이며, 규칙 기반 모드가 즉시 작동합니다

1단계: 저장소 클론

git clone https://github.com/your-org/ai-agent-taskpilot.git cd ai-agent-taskpilot

2단계: 환경 구성 (OpenAI 키는 선택 사항)

cp .env.example .env # 선택 사항: LLM 기반 계획을 위한 OpenAI 키 추가 OPENAI_API_KEY=sk-... # 규칙 기반 폴백을 사용하려면 비워두세요 AGENT_MAX_ITERATIONS=10 MEMORY_LONG_TERM_CAPACITY=500

3단계: Docker Compose로 모든 서비스 시작

docker compose up --build -d # 서비스 시작 중: # ✓ web-dashboard → http://localhost:3002 # ✓ dotnet-api → http://localhost:5002 # ✓ agent-core → http://localhost:8001

4단계: API를 통해 에이전트 작업 실행

curl -X POST http://localhost:5002/api/agent/run \ -H "Content-Type: application/json" \ -d '{"goal": "사용자 인증 구현을 위한 작업을 생성하고 높은 우선순위 지정"}' # 반환: {"sessionId": "sess_abc", "status": "running"} # ws://localhost:5002/hubs/agent 에서 이벤트 스트림 모니터링

5단계: http://localhost:3002에서 칸반 보드 열기

🔒 보안 고려사항

설정 가능한 가드레일을 갖춘 안전한 자율 에이전트 운영

🔑 API 키 보안

OpenAI API 키는 환경 변수에만 저장됩니다. Python 에이전트 코어는 키를 절대 로그하지 않습니다. OpenAI에 대한 요청은 서버 사이드에서만 이루어지며 — 키는 브라우저 클라이언트에 절대 노출되지 않습니다.

🔁 반복 제한

ReAct 루프에는 설정 가능한 최대 반복 횟수가 있습니다(기본값 10). 이는 LLM 기반 계획을 사용할 때 무한 루프와 과도한 토큰 비용을 방지합니다.

🛡 도구 접근 제어

각 에이전트 도구는 실행 전에 입력을 검증합니다. TaskManagerTool은 필드 수준 검증을 시행합니다. WebSearcherTool에는 도메인 허용/차단 목록이 있습니다.

📡 WebSocket 인증

SignalR 허브 연결은 유효한 세션 토큰이 필요합니다. 인증되지 않은 클라이언트는 공개 이벤트만 받습니다; 작업 수준 이벤트는 권한 부여가 필요합니다.

💾 메모리 프라이버시

장기 메모리 항목은 사용자 세션 범위로 지정됩니다. 민감한 작업의 임베딩은 외부 임베딩 서비스에 전송하지 않고 로컬에 저장됩니다.

📱 모바일 푸시 알림

Expo 푸시 토큰은 암호화하여 저장됩니다. 알림 페이로드에는 작업 ID만 포함되며 — 전체 작업 데이터는 알림 수신 후 클라이언트가 가져옵니다.
