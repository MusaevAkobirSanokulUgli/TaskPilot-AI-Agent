Engineers waste 60%+ of their time on repetitive project-management workflows: bug triage, status reporting, task assignment, and standup preparation. ReAct Loop — 6 Phases (max 10 iterations)

The agent runs a Reason → Plan → Act → Observe → Remember → Check cycle. Each iteration either completes the goal or feeds its observations back into the next reasoning step.

1
REASON — Planner.reason() → Thought
Analyzes the instruction combined with memory context and any previous observations. Produces a natural-language thought that frames the problem and identifies what information is still needed.
2
PLAN — Planner.plan() → Plan with ToolCalls
Converts the thought into a structured plan: goal statement + ordered steps + specific tool invocations with arguments.
3
ACT — AgentLoop._execute_tool() → Action results
Dispatches each planned tool call to the appropriate Tool class. Calls tool.execute(arguments) and collects the raw result payload.
4
OBSERVE — AgentLoop._create_observation()
Synthesizes the raw tool results into a concise natural-language observation that captures what happened and what was learned.
5
REMEMBER — MemoryManager.store_interaction()
Stores the interaction in short-term and working memory. Significant entries (high importance score) are promoted to long-term memory with embeddings.
6
CHECK — Planner.check_completion() → bool
Detects completion keywords in the last observation. When OPENAI_API_KEY is set, also calls the LLM for a semantic completion check. Returns True to exit the loop or False to iterate again.

Planner — Dual Mode

LLM Mode (GPT-4o-mini)

Activated when OPENAI_API_KEY is set. Uses GPT-4o-mini for open-ended reasoning and dynamic tool selection. Best for ambiguous or complex instructions.

Rule-Based Mode (Keyword Heuristics)

Works without any API key. Keyword routing:

  • triage/bug → bug triage workflow
  • report → report generation
  • assign → assignment workflow
  • status/standup → standup summary
  • create task → task creation
  • analyze/code → code analysis

3-Tier Memory System

Short-Term Memory
  • Capacity: 20 entries
  • Eviction: FIFO (oldest dropped)
  • Significant entries promoted to long-term
  • Retrieval weight: 1.5×
Working Memory
  • Capacity: unbounded
  • Active task state only
  • Format: "Action: X → Result: Y"
  • Cleared on task completion
  • Retrieval weight: 1.2×
Long-Term Memory
  • Capacity: 500 entries
  • 128-dim hash-based embeddings
  • Cosine similarity retrieval
  • Eviction: LRU

Agent State Machine

idle → thinking → planning → acting → observing → completed

Terminal states: error or stopped. The loop may cycle back from observing to thinking up to 10 times before forcing completion.

ReAct Tsikli — 6 Bosqich (maksimal 10 takrorlash)

Agent Mulohaza → Rejalashtirish → Harakat → Kuzatish → Eslab Qolish → Tekshirish tsiklini bajaradi. Har bir takrorlash maqsadni yakunlaydi yoki kuzatuvlarini keyingi mulohaza bosqichiga qaytaradi.

1
MULOHAZA — Planner.reason() → Fikr
Ko'rsatmani xotira konteksti va oldingi kuzatuvlar bilan birlashtirgan holda tahlil qiladi. Muammoni belgilaydigan va qanday ma'lumot kerakligini aniqlash uchun tabiiy til fikri yaratadi.
2
REJALASHTIRISH — Planner.plan() → Vosita Chaqiruvlari bilan Reja
Fikrni tuzilgan rejaga aylantiradi: maqsad bayoni + tartiblangan qadamlar + argumentlari bilan aniq vosita chaqiruvlari.
3
HARAKAT — AgentLoop._execute_tool() → Natijalar
Har bir rejalashtirilgan vosita chaqiruvini tegishli Tool sinfiga yuboradi. tool.execute(arguments) ni chaqiradi va xom natija yukini to'playdi.
4
KUZATISH — AgentLoop._create_observation()
Xom vosita natijalarini nima sodir bo'lgani va nima o'rganilgani haqida qisqacha tabiiy til kuzatuviga sintez qiladi.
5
ESLAB QOLISH — MemoryManager.store_interaction()
O'zaro ta'sirni qisqa muddatli va ish xotirasida saqlaydi. Muhim yozuvlar (yuqori ahamiyat ballari) embedding bilan uzoq muddatli xotiraga ko'tariladi.
6
TEKSHIRISH — Planner.check_completion() → bool
Oxirgi kuzatuvdagi yakunlash kalit so'zlarini aniqlaydi. OPENAI_API_KEY o'rnatilganda, semantik tekshirish uchun LLM ham chaqiriladi. Tsikldan chiqish uchun True yoki qayta takrorlash uchun False qaytaradi.

ReAct 루프 — 6단계 (최대 10회 반복)

에이전트는 추론 → 계획 → 행동 → 관찰 → 기억 → 확인 사이클을 실행합니다. 각 반복은 목표를 완료하거나 관찰 결과를 다음 추론 단계로 피드백합니다.

1
추론 — Planner.reason() → 사고
명령을 메모리 컨텍스트 및 이전 관찰 결과와 결합하여 분석합니다. 문제를 정의하고 필요한 정보를 식별하는 자연어 사고를 생성합니다.
2
계획 — Planner.plan() → 도구 호출 포함 계획
사고를 구조화된 계획으로 변환합니다: 목표 설명 + 순서가 있는 단계 + 인수가 포함된 특정 도구 호출.
3
행동 — AgentLoop._execute_tool() → 작업 결과
계획된 각 도구 호출을 적절한 Tool 클래스로 디스패치합니다. tool.execute(arguments)를 호출하고 원시 결과 페이로드를 수집합니다.
4
관찰 — AgentLoop._create_observation()
원시 도구 결과를 무슨 일이 일어났고 무엇을 배웠는지 포착하는 간결한 자연어 관찰로 합성합니다.
5
기억 — MemoryManager.store_interaction()
상호작용을 단기 및 작업 메모리에 저장합니다. 중요한 항목(높은 중요도 점수)은 임베딩과 함께 장기 메모리로 승격됩니다.
6
확인 — Planner.check_completion() → bool
마지막 관찰에서 완료 키워드를 감지합니다. OPENAI_API_KEY가 설정된 경우 의미론적 완료 확인을 위해 LLM도 호출합니다. 루프를 종료하려면 True, 다시 반복하려면 False를 반환합니다.

LLM 모드 (GPT-4o-mini)

규칙 기반 모드 (키워드 휴리스틱)

API 키 없이도 작동합니다. 키워드 라우팅:

  • triage/bug → 버그 분류 워크플로우
  • report → 보고서 생성
  • assign → 할당 워크플로우
  • status/standup → 스탠드업 요약
  • create task → 작업 생성
  • analyze/code → 코드 분석

3계층 메모리 시스템

단기 메모리
  • 용량: 20개 항목
  • 제거: FIFO (가장 오래된 것부터)
  • 중요한 항목은 장기 메모리로 승격
  • 검색 가중치: 1.5×
작업 메모리
  • 용량: 무제한
  • 활성 작업 상태만 저장
  • 형식: "작업: X → 결과: Y"
  • 작업 완료 시 초기화
  • 검색 가중치: 1.2×
장기 메모리
  • 용량: 500개 항목
  • 128차원 해시 기반 임베딩
  • 코사인 유사도 검색
  • 제거: LRU

에이전트 상태 기계

대기 → 추론 중 → 계획 중 → 행동 중 → 관찰 중 → 완료

종료 상태: 오류 또는 중지됨. 강제 완료 전까지 관찰 중에서 추론 중으로 최대 10번 순환할 수 있습니다.

웹 프론트엔드 페이지

대시보드

상태 게이지, 액션 타임라인, 빠른 명령 입력. SignalR을 통한 실시간 업데이트.

채팅

색상으로 구분된 단계가 포함된 ReAct 단계 시각화: 황색=추론, 보라색=계획, 파란색=행동, 녹색=관찰.

작업 (칸반)

4열 드래그 앤 드롭 보드: 할 일, 진행 중, 완료, 차단됨. 새 작업에 대한 우선순위 선택기.

메모리 뷰어

단기, 작업, 장기 메모리를 위한 3탭 뷰. 관련성 점수 및 타임스탬프 표시.

보고서

유형별 보고서 생성(상태/일간/주간/스탠드업/스프린트). 스타일링된 Markdown으로 렌더링.
